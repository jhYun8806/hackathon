# 제작 원칙 — LMS Inquiry Assistant

> 2026 제품실 AI 해커톤 작업 중 적용한 중요 기준 / 원칙 / 가이드

---

## 1. AI 답변 품질 원칙

**실제 담당자 말투를 모방하지 않는다**
- RAG로 가져온 사례는 맥락 참고용. 담당자의 구어체·특정 용어를 그대로 복사하지 않도록 system prompt에 명시
- 불확실하거나 사례에 없는 내용은 "담당 매니저에게 직접 문의 부탁드립니다"로 안내 → 오답보다 미답이 낫다

**인사는 첫 메시지에만**
- 멀티턴 대화에서 매 응답마다 인사하면 어색함 → `[첫 번째 질문]` / `[이어지는 질문]` 태그로 분기
- history 배열 길이로 판단, 프롬프트에 태그 삽입

**JSON 출력 강제**
- `{"answer":"...","followUps":[...]}` 포맷 고정 → 파싱 실패 시 raw 텍스트 폴백
- 스트리밍 시 부분 JSON에서 answer 값만 점진 추출 (정규식: `/"answer"\s*:\s*"((?:[^"\\]|\\.)*)/`)

## 2. 성능 원칙

**병렬 가능한 I/O는 반드시 병렬화**
- Supabase inquiry_cache 검색 + notices 검색 → `Promise.all`로 동시 실행
- 순차 실행 대비 ~200~400ms 단축

**체감 속도 = 실제 속도보다 중요**
- Claude 응답 전체를 기다렸다가 한 번에 렌더링하지 않음
- SSE(Server-Sent Events) 스트리밍으로 토큰 생성 즉시 클라이언트에 전달
- 타이핑 인디케이터는 첫 chunk 수신 전까지만 표시

## 3. UI/UX 원칙

**모달 height 고정**
- 뷰(홈 ↔ 채팅) 전환 시 height가 변하면 레이아웃 점프로 불쾌감 → `height: 800px` 고정

**채팅 이력은 모달 닫아도 유지**
- 홈 화면으로 돌아가도 `msgs` / `history` state 보존
- "이전 대화 이어보기" 버튼으로 재진입 가능

**관련 링크는 버블 안에 포함**
- AI 답변 버블 밖에 따로 카드 형태로 분리하면 연결고리가 끊김
- 버블 하단 구분선 아래에 인라인으로 표시

**후속 질문 버튼은 링크와 시각적으로 구분**
- 빨간색 → 사용자가 링크로 오인 → 회색 계열 버튼 + `Q.` 접두사로 변경

**캐릭터 애니메이션은 채팅창 내부에서만**
- 헤더 르탄이 아이콘: float 애니메이션 없음 (브랜딩용, 시선 분산 방지)
- 채팅 메시지 아바타: float 애니메이션 유지 (생동감)
- FAB 위 캐릭터: 머리만 까딱 (몸통 고정, CSS `clip-path: inset()` 레이어 분리 + 머리 레이어만 rotation)

**답변 완료 후에만 액션 버튼 노출**
- 스트리밍 도중 "해결됐어요 / 문의 등록하기" 버튼이 보이면 사용자가 답변이 끝났다고 착각
- AI 메시지 타입에 `complete: boolean` 필드 추가 → SSE `done` 이벤트 수신 시 `true` 전환
- 버튼 등장 시 `buttonReveal` 애니메이션 (0.45s 슬라이드업) 적용으로 자연스럽게 떠오름

**FAB 관련 요소는 단일 래퍼로 묶어 위치 일관성 확보**
- 말풍선·FabRtani 캐릭터·FAB 버튼을 각각 독립 `fixed` 배치하면 스크롤 시 서로 어긋남
- 단일 `fixed` 래퍼 (`bottom: 30px`, `right: 1.7rem`, `gap: 30px`)에 묶어 항상 함께 이동

## 4. 코드 구조 원칙

**순수 함수는 별도 파일로 분리해 테스트 가능하게**
- `lib/utils.ts` — 외부 의존성 없는 순수 함수만 모음
- ChatBot.tsx / route.ts 양쪽에서 동일 로직 중복 정의하지 않음

**테스트는 외부 의존성 없이**
- Supabase / Anthropic API mock 없이 순수 로직만 테스트
- 실제 API 연동 테스트는 수동 검증으로 대체 (해커톤 범위)

**컴포넌트는 필요한 추상화만**
- `RtaniAvatar`, `StudentAvatar`, `FabRtani` 등 재사용 단위만 분리
- 한 번만 쓰는 로직은 인라인으로 유지

## 5. 데이터 원칙

**있는 데이터를 최대한 활용**
- 채널톡 Open API가 워크플로 구조를 지원하지 않음 → CDP(Chrome DevTools Protocol)로 UI DOM 파싱하여 FAQ 10건 수동 수집
- round_id가 Redash 데이터에 없을 때 → FALLBACK_ROUND_ID로 폴백, 서비스 중단 없이 처리

**캐시 TTL 1시간, 백그라운드 갱신**
- 매 요청마다 Redash 호출하면 응답 지연 → sync_meta로 마지막 동기화 시간 추적
- 오래됐으면 fire-and-forget 방식으로 백그라운드 갱신 (현재 요청 블로킹 없음)

**프로토타입에서 RLS는 비활성화**
- 해커톤 범위에서 빠른 개발 우선 → Supabase RLS 전체 비활성화
- 프로덕션 전 anon 정책 + RLS 설정 필요 (보완 항목으로 명시)
